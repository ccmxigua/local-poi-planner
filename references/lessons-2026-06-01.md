# Lessons / 踩坑记录 — 2026-06-01

本次用 local-poi-planner 帮用户搜索"附近 8km 内的 4D/4DX 电影院"时踩到的坑。

## 1. 网络搜索必须走 unified-search.sh，禁止 DIY curl/scrape

**现象**：DuckDuckGo Lite 返回空或垃圾；curl 百度/sogou/360 返回 HTML 且格式不一，解析极不稳定；Tavily research 脚本被 SIGKILL。

**根因**：
- DuckDuckGo Lite 对中文 query 不友好
- 百度/搜狗/360 有反爬机制（cookie、验证码、编码问题）
- Tavily research (`scripts/research.mjs`) 超时被系统 kill（SIGKILL），不是 HTTP 超时而是进程级别

**结论**：**所有 external web enrichment 必须走 `unified-search.sh`（deep search-layer 路径）。**
- 入口：`bash <unified-search>/scripts/unified-search.sh "<query>"`
- 对中文本地 query 实际只有 Grok（`grok-4.20-multi-agent-xhigh`）有效
- 不要用 curl、requests、playwright 或任何自建 HTTP 爬虫做中文搜索
- 如果 unified-search 返回的信息不够细粒度（如查不到某个影院是否有 4DX），应如实告知用户局限，而非转而自建爬虫

## 2. 高德 POI API 无法区分影院屏幕类型（4D/IMAX/Dolby）

**现象**：
- `amap_poi.search(keyword="4DX")` → 0 条
- `amap_poi.search(keyword="MX4D")` → 0 条
- `amap_poi.search(keyword="4D影厅")` → 40 条，但匹配的是"影厅"而非真 4D
- 高德 place/text API 同样 0 条

**根因**：高德 POI 数据中影院没有"4DX/IMAX/杜比"这类屏幕特性字段。`typecode=080600`（电影院）是最高粒度。

**结论**：查询影院屏幕特性（4D/IMAX/Dolby/ScreenX/CINITY）需要第二数据源：
- 猫眼 API（`m.maoyan.com`）→ 见下文第 3 条
- 或 unified-search 搜「XX影院 4DX」获取网页证据
- 或在报告中标注"POI 数据不包含屏幕类型，请到猫眼/淘票票确认"

## 3. 猫眼 API 查影院屏幕类型的方法

**关键发现**：
- 天津 cityId = **40**（不是 10，10 是上海）
- API: `GET https://m.maoyan.com/ajax/cinemaList?cityId=40&offset=0&limit=100`
- 屏幕类型在 **`tag.hallTypeVOList[].name`**，不是顶层的 `hallTypes`（顶层始终 null）
- 天津共 120 家影院，其中 **只有 6 家有 4D/4DX/MX4D 影厅**

**正确的 4D 检测代码**：
```python
for c in cinemas:
    hall_types = [h.get('name','') for h in (c.get('tag',{}).get('hallTypeVOList',[]) or [])]
    if any(t in h for t in ['4DX', '4D', 'MX4D', 'D-BOX'] for h in hall_types):
        # this cinema has 4D
```

**天津 4D 影院完整列表**（2026-06-01）：
| 影院 | 距离用户 | 4D 类型 |
|------|---------|---------|
| 万达影城（河东激光IMAX店） | 7.2km | 4DX厅 + 4D厅 |
| 万达影城（西青社会山IMAX店） | 19.1km | 4DX厅 |
| 福地国际影城（贻成福地广场店） | 34.4km | 4D厅 + MX4D厅 |
| 中影南方影城（金元宝东方广场店） | 34.6km | 4D厅 |
| 万达影城（滨海CINITY店） | 35.7km | 4DX厅 |
| 万达影城（武清威尼都XLAND店） | 41.8km | 4DX厅 |

**8km 内只有 1 家**：万达影城（河东激光IMAX店）= Amap 显示为"万达影城(天津河东万达广场店)"。

**注意**：CGV影城(梅江ScreenX店) 在 6.2km 但有 ScreenX 但无 4D——不能假设 CGV 一定有 4DX。

## 4. .env.local 加载陷阱

**现象**：`from scripts.amap_poi import search` 后 `search()` 返回 0 条结果。

**根因**：`amap_poi.py` 的 `_load_key()` 只做 `os.getenv("AMAP_KEY")`，不主动加载 `.env.local`。`.env.local` 在 skill 根目录，`amap_poi.py` 在 `scripts/` 子目录。

**正确做法**：在 `planner.py` 或调用脚本顶部显式加载：
```python
env_file = Path(__file__).resolve().parent.parent / '.env.local'
if env_file.exists():
    with open(env_file) as f:
        for line in f:
            line = line.strip()
            if line and not line.startswith('#') and '=' in line:
                k, v = line.split('=', 1)
                os.environ.setdefault(k.strip(), v.strip())
```
（`planner.py` 已有此逻辑，但直接 `import amap_poi` 再调用的方式不会自动加载。当前 skill 集成系统在调用前会通过 skill 加载器注入环境变量，但直接用 Python import 需要手动处理。）

## 5. macOS CoreLocation 自动化被系统权限弹窗阻塞

**现象**：所有程序化获取 GPS 坐标的尝试都失败。

**尝试的方法及结果**：
- CoreLocationCLI: 未安装
- Swift CoreLocation: 进程被权限弹窗 kill
- ipinfo.io: 返回城市级坐标（~10km 误差），不是 GPS 精度
- Playwright geolocation: `http://` 下报"Only secure origins are allowed"；`https://amap.com` 超时
- Python objc (pyobjc): 模块未安装

**结论**：macOS 的 Location Services 权限弹窗无法在自动化中点击。当前只能依赖：
- 用户在 Telegram 发送 GPS 坐标（推荐首选）
- 高德 IP 定位（`amap_ip_location.py`，城市级精度，~8km 误差）
- ipinfo.io（同样是城市级）

在报告中应明确标注坐标来源和精度级别。

## 6. Maoyan 距离 vs 高德距离

猫眼返回的 `distance` 字段是基于用户手机 GPS 的直线距离，而高德返回的是路径距离或直线距离（取决于 API）。两者可能不一致，交叉引用时注意标注来源。

## 7. unified-search deep mode 的 `--source` 参数被忽略

已确认：deep path（`run-search-layer.sh` → `search.py`）忽略 `--source` 参数，即使指定 `--source grok` 也会跑所有引擎。这是已知 bug，未修。查询端实际有效的引擎由 `~/.openclaw/credentials/search.json` 配置决定。

## 8. planner 静默执行 + 品类回退导致 query 语义全歪

**现象**：
- planner 全程 `subprocess.run(capture_output=True)`，零进度打印
- AI 等 2 分钟后看到"no output yet"，误判为卡死，kill 了进程
- 实际上 planner 正常需要 3-7 分钟，统一搜索的 `timeout=900`（15 分钟）完全够

**另一个现象**：
- 用户传入 `--category "4D电影院"`，但 CATEGORY_ALIASES 没有 `"4D电影院"` → `"电影院"` 映射
- normalize_category 保持了 `"4D电影院"`（raw 不为空）
- CATEGORY_KEYWORDS.get("4D电影院", ...) 未命中 → 回退到 CATEGORY_KEYWORDS["restaurant"]
- query 变成 `"双港 餐厅 堂食 有座位 4DX/MX4D/4D影厅/D-BOX"`——语义全歪，搜出来的全是饭馆

**修复**：
1. planner.py `main()` 加了 6 个进度 `print(..., flush=True)` + 耗时
2. planner.py `CATEGORY_ALIASES` 加了 `"4D电影院": "电影院"`
3. SKILL.md 加了 planner 静默执行时间说明（3-10 分钟正常，不要因无输出 kill）

**教训**：
- `capture_output=True` 彻底吞掉了所有进度，对长任务必须加中间 print
- 品类别名系统缺少对"修饰词+品类"模式的处理（"4D电影院""快捷酒店""精酿啤酒馆"……），目前只能逐条硬编码
- 需要给 build_queries 加一个日志 print，这样下次出现语义错误可以即时发现
